AI-generated code requires **proportional verification** based on risk level. Blindly accepting all output or paranoidly reviewing every line both waste time. This tutorial helps you calibrate your trust and build an efficient verification strategy.

---

## The Problem: Verification Debt

Research consistently shows that AI-generated code has higher defect rates than human-written code:

| Metric | AI vs Human | Source |
|--------|-------------|--------|
| Logic errors | 1.75x more | ACM study, 2025 |
| Security flaws | 45% contain vulnerabilities | Veracode GenAI Report, 2025 |
| XSS vulnerabilities | 2.74x more | CodeRabbit study, 2025 |
| PR size increase | +18% | Jellyfish, 2025 |
| Incidents per PR | +24% | Cortex.io, 2026 |
| Change failure rate | +30% | Cortex.io, 2026 |

<Aside type="note">
**Key insight**: AI produces code faster, but verification becomes the bottleneck. The question is not "does it work?" but "how do I know it works?"
</Aside>

<Aside type="tip">
**Nuance on downstream maintainability**: A 2-phase blind RCT (Borg et al., 2025, n=151 professional developers) found no significant difference in the time needed for downstream developers to evolve AI-generated vs. human-generated code. The defect rates above are real, but they do not systematically translate into higher maintenance burden for the next developer. The risk is more narrowly scoped than commonly assumed.
</Aside>

## The Verification Spectrum

Not all code needs the same scrutiny. Match verification effort to risk:

<CardGrid>
  <Card title="Boilerplate" icon="approve-check-circle">
    **Verification:** Light skim

    Configs, imports, etc. Spend 10-30 seconds glancing at it. Trust the structure.
  </Card>
  <Card title="Utility Functions" icon="pencil">
    **Verification:** Quick test

    Formatters, helpers, etc. Spend 1-2 minutes with one happy-path test.
  </Card>
  <Card title="Business Logic" icon="open-book">
    **Verification:** Deep review + tests

    Spend 5-15 minutes with line-by-line review and edge case coverage.
  </Card>
  <Card title="Security-Critical" icon="warning">
    **Verification:** Maximum + tools

    Auth, crypto, input validation. Spend 15-30 minutes with static analysis, fuzzing, and peer review.
  </Card>
</CardGrid>

| Code Type | Verification Level | Time Investment | Techniques |
|-----------|-------------------|-----------------|------------|
| Boilerplate (configs, imports) | Light skim | 10-30 sec | Glance, trust structure |
| Utility functions (formatters, helpers) | Quick test | 1-2 min | One happy-path test |
| Business logic | Deep review + tests | 5-15 min | Line-by-line, edge cases |
| Security-critical (auth, crypto, validation) | Maximum + tools | 15-30 min | Static analysis, fuzzing, peer review |
| External integrations (APIs, databases) | Integration tests | 10-20 min | Mock + real endpoint test |

## Solo vs Team Verification

<Tabs>
  <TabItem label="Solo Developer">
    Without peer reviewers, compensate with:

    <Steps>

    1. **High test coverage (&gt;70%)**: Your safety net

    2. **Vibe Review**: An intermediate layer between "accept blindly" and "review every line"
       - Read the commit message or summary
       - Skim the diff for unexpected file changes
       - Run the tests
       - Quick sanity check in the app
       - Ship if green

    3. **Static analysis tools**: ESLint, SonarQube, Semgrep catch what you miss

    4. **Time-boxing**: Do not spend 30 minutes reviewing a 10-line utility

    </Steps>

    ```
    Solo workflow:
    Generate → Vibe Review → Tests pass? → Ship
                    ↓
            Tests fail? → Deep review → Fix
    ```
  </TabItem>
  <TabItem label="Team Strategy">
    With multiple developers:

    <Steps>

    1. **AI first-pass review**: Let Claude or Copilot review first (catches 70-80% of issues)

    2. **Human sign-off required**: AI review does not equal approval

    3. **Domain experts for critical paths**: Security code goes to security-trained reviewers

    4. **Rotate reviewers**: Prevent blind spots from forming

    </Steps>

    ```
    Team workflow:
    Generate → AI Review → Human Review → Merge
                  ↓              ↓
             Flag issues    Final approval
    ```
  </TabItem>
</Tabs>

## The "Prove It Works" Checklist

Before shipping AI-generated code, verify each category:

<Tabs>
  <TabItem label="Functional Correctness">
    - Happy path works (manual test or automated)
    - Edge cases handled (null, empty, boundary values)
    - Error states are graceful (no silent failures)
  </TabItem>
  <TabItem label="Security Baseline">
    - Input validation present (never trust user input)
    - No hardcoded secrets (grep for `password`, `secret`, `key`)
    - Auth/authz checks intact (did not bypass existing guards)
  </TabItem>
  <TabItem label="Integration Sanity">
    - Existing tests still pass
    - No unexpected file changes in the diff
    - Dependencies added are justified and audited
  </TabItem>
  <TabItem label="Code Quality">
    - Follows project conventions (naming, structure)
    - No obvious performance issues (N+1 queries, memory leaks)
    - Comments explain "why" not "what"
  </TabItem>
</Tabs>

## Anti-Patterns to Avoid

| Anti-Pattern | Problem | Better Approach |
|--------------|---------|-----------------|
| "It compiles, ship it" | Syntax correctness does not equal logic correctness | Run at least one test |
| "AI wrote it, must be secure" | AI optimizes for plausible, not safe | Always review security-critical code manually |
| "Tests pass, done" | Tests might not cover the changed lines | Check test coverage of modified lines |
| "Same as last time" | Context changes; AI may generate different code | Each generation is independent |
| "Senior dev wrote the prompt" | Seniority does not guarantee output quality | Review output, not input |
| "It's just boilerplate" | Even boilerplate can hide issues | At minimum, skim for surprises |

## Calibrating Over Time

Your verification strategy should evolve:

<Steps>

1. **Start cautious**

   Review everything when you are new to Claude Code

2. **Track failure patterns**

   Where do bugs slip through? Record the patterns

3. **Tighten critical paths**

   Double down on areas with past incidents

4. **Relax low-risk areas**

   Trust AI more for stable, well-tested code types

5. **Periodic audits**

   Spot-check "trusted" code occasionally

</Steps>

<Aside type="tip">
**Mental model**: Think of AI as a capable junior developer. You would not deploy their code unreviewed, but you also would not rewrite everything they produce.
</Aside>
