## Session Search & Resume

### Native Commands

| Command | Use Case |
|---------|----------|
| `claude -c` / `claude --continue` | Resume most recent session |
| `claude -r <id>` / `claude --resume <id>` | Resume specific session by ID |
| `claude --resume` | Interactive session picker |

Sessions are stored locally at `~/.claude/projects/<project>/` as JSONL files.

### Community Tools Comparison

| Tool | Install | Speed | Dependencies | Resume Support |
|------|---------|-------|--------------|----------------|
| **session-search.sh** | Copy script | 10ms list / 400ms search | None (bash) | Displayed |
| claude-conversation-extractor | `pip install` | 230ms / 1.7s | Python | No |
| claude-code-transcripts | `uvx` | N/A | Python | No |
| ran CLI | `npm -g` | Fast | Node.js | No |

### Recommended: session-search.sh

Zero-dependency bash script optimized for speed with ready-to-use resume commands.

**Install:**
```bash
cp examples/scripts/session-search.sh ~/.claude/scripts/cs
chmod +x ~/.claude/scripts/cs
echo "alias cs='~/.claude/scripts/cs'" >> ~/.zshrc
source ~/.zshrc
```

**Usage:**
```bash
cs                          # List 10 most recent sessions (~15ms)
cs "authentication"         # Single keyword search (~400ms)
cs "Prisma migration"       # Multi-word AND search
cs -n 20                    # Show 20 results
cs -p myproject "bug"       # Filter by project name
cs --since 7d               # Sessions from last 7 days
cs --since today            # Today's sessions only
cs --json "api" | jq .      # JSON output for scripting
```

**Example output:**
```
2026-01-15 08:32 | my-project       | Implement OAuth flow for...
  claude --resume 84287c0d-8778-4a8d-abf1-eb2807e327a8

2026-01-14 21:13 | other-project    | Fix database migration...
  claude --resume 1340c42e-eac5-4181-8407-cc76e1a76219
```

Copy-paste the `claude --resume` command to continue any session.

### Session Resume Limitations

<Aside type="caution">
Native `--resume` is directory-scoped. Claude Code stores sessions at `~/.claude/projects/<encoded-path>/` where `<encoded-path>` is derived from your project's absolute path. Moving your project folder breaks resume.
</Aside>

**Manual migration when moving a project:**

```bash
# Before moving project
cd ~/.claude/projects/
ls -la  # Note the current encoded path

# Move your project
mv /old/location/myapp /new/location/myapp

# Rename session directory to match new path
cd ~/.claude/projects/
mv -- -old-location-myapp- -new-location-myapp-

# Verify
cd /new/location/myapp
claude --continue  # Should resume successfully
```

## Setting Up Session Logging

<Steps>
1. **Install the Logger Hook**

   ```bash
   mkdir -p ~/.claude/hooks
   cp examples/hooks/bash/session-logger.sh ~/.claude/hooks/
   chmod +x ~/.claude/hooks/session-logger.sh
   ```

2. **Register in Settings**

   Add to `~/.claude/settings.json`:

   ```json
   {
     "hooks": {
       "PostToolUse": [
         {
           "type": "command",
           "command": "~/.claude/hooks/session-logger.sh"
         }
       ]
     }
   }
   ```

3. **Verify Installation**

   ```bash
   ls ~/.claude/logs/
   # Should see: activity-2026-01-14.jsonl

   tail -5 ~/.claude/logs/activity-$(date +%Y-%m-%d).jsonl | jq .
   ```
</Steps>

### Configuration Options

| Environment Variable | Default | Description |
|---------------------|---------|-------------|
| `CLAUDE_LOG_DIR` | `~/.claude/logs` | Where to store logs |
| `CLAUDE_LOG_TOKENS` | `true` | Enable token estimation |
| `CLAUDE_SESSION_ID` | auto-generated | Custom session identifier |

### Reading Logs for Configuration Quality

Token counts tell you how much you used Claude Code. JSONL logs also reveal **how well your configuration is working**.

Three patterns signal that a skill, rule, or CLAUDE.md section needs updating:

<Tabs>
  <TabItem label="Repeated File Reads">
    If Claude reads the same file 3+ times in one session, the content it needs probably is not where it expects.

    ```bash
    # Files read more than 3x in recent sessions
    jq -r 'select(.tool == "Read") | .file' ~/.claude/logs/activity-*.jsonl \
      | sort | uniq -c | sort -rn | awk '$1 > 3'
    ```

    **Fix**: Move relevant context into a skill or CLAUDE.md section.
  </TabItem>
  <TabItem label="Repeated Tool Failures">
    A Bash command that fails repeatedly across sessions usually means a skill has an outdated path or renamed binary.

    ```bash
    # Failing commands
    jq -r 'select(.tool == "Bash" and (.exit_code // 0) != 0) | .command' \
      ~/.claude/logs/activity-*.jsonl | sort | uniq -c | sort -rn | head -10
    ```

    **Fix**: Update the failing command in your skill or CLAUDE.md.
  </TabItem>
  <TabItem label="High Edit Frequency">
    Files edited heavily across sessions often indicate missing context about the file's purpose or conventions.

    ```bash
    # Most-edited files
    jq -r 'select(.tool == "Edit") | .file' ~/.claude/logs/activity-*.jsonl \
      | sort | uniq -c | sort -rn | head -10
    ```

    **Fix**: Document conventions around that file in CLAUDE.md.
  </TabItem>
</Tabs>

### Log Format

Each log entry is a JSON object:

```json
{
  "timestamp": "2026-01-14T15:30:00Z",
  "session_id": "1705234567-12345",
  "tool": "Edit",
  "file": "src/components/Button.tsx",
  "project": "my-app",
  "tokens": {
    "input": 350,
    "output": 120,
    "total": 470
  }
}
```

## Cost Tracking

### Token Estimation

The logger estimates tokens using a simple heuristic: **~4 characters per token**. This is approximate and tends to slightly overestimate.

### Cost Rates

Default rates are for Claude Sonnet. Adjust via environment variables:

```bash
# Sonnet rates (default)
export CLAUDE_RATE_INPUT=0.003   # $3/1M tokens
export CLAUDE_RATE_OUTPUT=0.015  # $15/1M tokens

# Opus rates
export CLAUDE_RATE_INPUT=0.015   # $15/1M tokens
export CLAUDE_RATE_OUTPUT=0.075  # $75/1M tokens

# Haiku rates
export CLAUDE_RATE_INPUT=0.00025 # $0.25/1M tokens
export CLAUDE_RATE_OUTPUT=0.00125 # $1.25/1M tokens
```

### Budget Alerts

Add to your shell profile for daily budget warnings:

```bash
# ~/.zshrc or ~/.bashrc
claude_budget_check() {
  local cost=$(session-stats.sh --json 2>/dev/null | jq -r '.summary.estimated_cost.total // 0')
  local threshold=5.00  # $5 daily budget

  if (( $(echo "$cost > $threshold" | bc -l) )); then
    echo "Warning: Claude Code daily spend: \$$cost (threshold: \$$threshold)"
  fi
}

# Run on shell start
claude_budget_check
```

## Activity Monitoring

### Session JSONL: The Ground Truth

Every tool call Claude Code makes is recorded in session JSONL files at `~/.claude/projects/<project>/`. Each entry with `type: "assistant"` contains a `content` array where `type: "tool_use"` blocks document every action.

```bash
# Inspect tool calls in a session
cat ~/.claude/projects/-your-project-/SESSION_ID.jsonl | \
  jq 'select(.type == "assistant") | .message.content[]? | select(.type == "tool_use") | {tool: .name, input: .input}'
```

### What Tool Calls Reveal

| Tool | What It Exposes |
|------|----------------|
| `Read` | Files accessed (path, line range) |
| `Write` / `Edit` | Files modified (path, content delta) |
| `Bash` | Commands executed (full command string) |
| `WebFetch` | URLs fetched (may include data sent in POST) |
| `Task` | Subagent spawns (prompt passed to sub-model) |
| `Glob` / `Grep` | Search patterns and scope |

### Practical Audit Queries

```bash
SESSION=~/.claude/projects/-your-project-/SESSION_ID.jsonl

# All files read in a session
jq 'select(.type == "assistant") | .message.content[]? | select(.type == "tool_use" and .name == "Read") | .input.file_path' "$SESSION"

# All bash commands executed
jq 'select(.type == "assistant") | .message.content[]? | select(.type == "tool_use" and .name == "Bash") | .input.command' "$SESSION"

# All URLs fetched
jq 'select(.type == "assistant") | .message.content[]? | select(.type == "tool_use" and .name == "WebFetch") | .input.url' "$SESSION"

# Count tool usage by type
jq -r 'select(.type == "assistant") | .message.content[]? | select(.type == "tool_use") | .name' "$SESSION" | sort | uniq -c | sort -rn
```

### Sensitive Patterns to Watch

| Pattern | Risk | Detection |
|---------|------|-----------|
| `Read` on `.env`, `*.pem`, `id_rsa` | Credential access | Filter for `\.(env\|pem\|key)$` in file paths |
| `Bash` with `rm -rf`, `git push --force` | Destructive action | Filter for dangerous command patterns |
| `WebFetch` on external URLs | Data exfiltration risk | Audit all fetched URLs |
| `Write` on files outside project root | Scope creep | Check paths against working directory |

<Aside>
Claude Code operates read-write on your filesystem with your user permissions. The JSONL audit trail is your record of what happened. For teams, consider syncing these logs to immutable storage.
</Aside>

## External Monitoring Tools

| Tool | Type | What It Does | Install |
|------|------|-------------|---------|
| **ccusage** | CLI / TUI | Cost tracking from JSONL. ~10K GitHub stars | `npm i -g ccusage` |
| **claude-code-otel** | OpenTelemetry | Emits spans to any OTEL collector | `npm i -g claude-code-otel` |
| **Akto** | SaaS / self-hosted | API security guardrails + audit trail | [akto.io](https://akto.io) |
| **MLflow Tracing** | CLI + SDK | Exact token counts, tool spans, LLM-as-judge | `pip install mlflow` |
| **ccboard** | TUI + Web | Unified dashboard for sessions, costs, stats | `cargo install ccboard` |

### Decision Guide

```
Want cost numbers fast?          → ccusage (CLI, 0 config)
Need enterprise audit trail?     → claude-code-otel + Grafana or Akto
Already using MLflow for ML?     → MLflow tracing integration
Need agent regression detection? → MLflow tracing + LLM-as-judge
Want a persistent TUI/Web UI?    → ccboard
```

### ccusage

```bash
npm i -g ccusage
ccusage          # Today's usage
ccusage --days 7 # Last 7 days
```

Reads directly from `~/.claude/projects/**/*.jsonl`. No API keys, no data sent externally.

### claude-code-otel

```bash
npm i -g claude-code-otel
claude-code-otel --collector http://localhost:4318
```

Spans include tool name, duration, token counts. Plug into any OTEL-compatible backend (Jaeger, Tempo, Datadog).

### MLflow Tracing

<Aside type="tip">
MLflow intercepts at the API level, not post-hoc from JSONL. It captures **exact** token counts (vs ~15-25% variance of hook-based estimation) and enables **LLM-as-judge** regression detection.
</Aside>

#### CLI Mode (No Python Required)

```bash
pip install "mlflow[genai]>=3.4"

# Enable tracing
mlflow autolog claude

# With custom backend
mlflow autolog claude -u sqlite:///mlflow.db

# Check status / disable
mlflow autolog claude --status
mlflow autolog claude --disable
```

Launch the UI:
```bash
mlflow server  # http://localhost:5000
```

**What gets captured**: user prompts, assistant responses, tool calls (name + inputs + outputs), token counts (exact), latency per call, session metadata.

#### MCP Server: Bidirectional Integration

Claude Code can query its own traces directly:

```json
{
  "mcpServers": {
    "mlflow-mcp": {
      "command": "uv",
      "args": ["run", "--with", "mlflow[mcp]>=3.5.1", "mlflow", "mcp", "run"],
      "env": { "MLFLOW_TRACKING_URI": "<your-tracking-uri>" }
    }
  }
}
```

Once configured, you can ask Claude Code: *"Find all sessions where the backend-architect agent used more than 20 tool calls"* -- it queries MLflow directly.

## Proxying Claude Code

<Aside>
A common question: "Can I use Proxyman/Charles to see what Claude Code sends to Anthropic?" Short answer: not directly. Claude Code is a Node.js process that ignores system-level proxy settings.
</Aside>

### Available Options

<Tabs>
  <TabItem label="MITM Certificate">
    Force Node.js to trust your proxy's CA certificate:

    ```bash
    export NODE_EXTRA_CA_CERTS="/path/to/proxyman-ca.pem"
    claude
    ```

    **Caveat**: Some Claude Code versions use certificate pinning -- this may still fail.
  </TabItem>
  <TabItem label="API URL Redirect">
    Point Claude Code at a local interceptor:

    ```bash
    export ANTHROPIC_API_URL="http://localhost:8080"
    claude
    ```

    Run any HTTP proxy on port 8080 that forwards to `https://api.anthropic.com`.
  </TabItem>
  <TabItem label="mitmproxy (Recommended)">
    ```bash
    brew install mitmproxy
    mitmproxy --listen-port 8080

    # In a new terminal
    export NODE_EXTRA_CA_CERTS="$(python3 -c 'import mitmproxy.certs; print(mitmproxy.certs.Cert.default_ca_path())')"
    export HTTPS_PROXY="http://localhost:8080"
    claude
    ```

    The mitmproxy web UI at `http://localhost:8081` shows full request/response bodies.
  </TabItem>
</Tabs>

<Aside type="caution">
Proxied traffic includes everything in the conversation context -- file contents, your code, any secrets Claude encountered. Handle proxy logs accordingly.
</Aside>

## Patterns & Best Practices

### Weekly Review

```bash
session-stats.sh --range week
```

Look for:
- Unusually high token usage days
- Repeated operations on same files (inefficiency signal)
- Project distribution (where time is spent)

### Per-Project Tracking

```bash
export CLAUDE_SESSION_ID="project-myapp-$(date +%s)"
claude
```

### Team Aggregation

```bash
# Sync to S3 daily
aws s3 sync ~/.claude/logs/ s3://company-claude-logs/$(whoami)/

# Combine and analyze
aws s3 sync s3://company-claude-logs/ /tmp/team-logs/
cat /tmp/team-logs/*/activity-$(date +%Y-%m-%d).jsonl | \
  jq -s 'group_by(.project) | map({project: .[0].project, total_tokens: [.[].tokens.total] | add})'
```

### Log Rotation

```bash
# Clean logs older than 30 days
find ~/.claude/logs -name "*.jsonl" -mtime +30 -delete
```

## Limitations

### What Monitoring Cannot Do

| Limitation | Reason |
|------------|--------|
| Exact token counts | CLI does not expose API token metrics |
| TTFT (Time to First Token) | Hook runs after tool completes |
| Real-time streaming metrics | No hook event during response generation |
| Actual API costs | Token estimates are heuristic |
| Model selection per request | Log does not capture which model was used |
| Context window usage | No visibility into current context percentage |

### What You Can Trust

| Metric | Accuracy |
|--------|----------|
| Tool usage counts | Exact count of each tool invocation |
| File access patterns | Which files were touched |
| Relative comparisons | Day-to-day and project-to-project trends |
| Operation timing | When tools were used (timestamp) |
| Token estimates | ~15-25% variance from actual billing |
