## Effort Levels

The `effort` parameter (Opus 4.6 API) controls the model's **overall computational budget** — not just thinking tokens, but tool calls, verbosity, and analysis depth.

<Tabs>
  <TabItem label="low">
    **Mechanical, no design decisions needed**

    > `"Rename getUserById to findUserById across src/"` — Find-replace scope, zero reasoning required.
  </TabItem>
  <TabItem label="medium">
    **Clear pattern, defined scope, one concern**

    > `"Convert fetchUser() in api/users.ts from callbacks to async/await"` — Pattern is known, scope bounded.
  </TabItem>
  <TabItem label="high">
    **Design decisions, edge cases, multiple concerns**

    > `"Redesign error handling in the payment module: add retry logic, partial failure recovery, and idempotency guarantees"` — Architectural choices, not just pattern application.
  </TabItem>
  <TabItem label="max">
    **Cross-system reasoning, irreversible decisions** (Opus 4.6 only)

    > `"Analyze the microservices event pipeline for race conditions across order-service, inventory-service, and notification-service"` — Multi-service hypothesis testing, adversarial thinking.
  </TabItem>
</Tabs>

Assign models to agents based on **role**, not importance:

  </Card>
  <Card title="Architecture Reviewer">
    **Model**: Opus

    ```yaml

    ```
  </Card>

<Aside type="caution">
Haiku is for mechanical tasks only. If the implementation requires design decisions or complex business logic, use Sonnet — state this in the task prompt.
</Aside>

<Aside type="tip">
Add a model reminder to your CLAUDE.md:
```
# Model reminder
Default: Sonnet. Haiku for mechanical tasks. Opus for architecture and security audits.
```
</Aside>

## Understanding Claude's Mental Model

Understanding how Claude Code "thinks" makes you more effective.

### Claude's View of Your Project

```
┌─────────────────────────────────────────────────────────┐
│                   YOUR PROJECT                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌─────────────┐    ┌─────────────┐    ┌───────────┐   │
│   │   Files     │    │   Git       │    │  Config   │   │
│   │   (.ts,.py) │    │   History   │    │  Files    │   │
│   └─────────────┘    └─────────────┘    └───────────┘   │
│          │                  │                  │        │
│          ▼                  ▼                  ▼        │
│   ┌─────────────────────────────────────────────────┐   │
│   │              Claude's Understanding             │   │
│   │   - File structure & relationships              │   │
│   │   - Code patterns & conventions                 │   │
│   │   - Recent changes (from git)                   │   │
│   │   - Project rules (from CLAUDE.md)              │   │
│   └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### What Claude Knows vs. Doesn't Know

<Tabs>
  <TabItem label="What Claude Knows">
    | Capability | Description |
    |------------|-------------|
    | **File Structure** | Can navigate and search your files |
    | **Code Content** | Can read and understand code |
    | **Git State** | Sees branches, commits, changes |
    | **Project Rules** | Reads CLAUDE.md for conventions |
  </TabItem>
  <TabItem label="What Claude Doesn't Know">
    | Limitation | Description |
    |------------|-------------|
    | **Runtime State** | Can't see running processes |
    | **External Services** | Can't access your databases directly |
    | **Your Intent** | Needs clear instructions |
    | **Hidden Files** | Respects .gitignore by default |
  </TabItem>
</Tabs>

<Aside type="caution" title="Pattern Amplification">
Claude mirrors the patterns it finds. In well-structured codebases, it produces consistent, idiomatic code. In messy codebases without clear abstractions, it perpetuates the mess. If your code lacks good patterns, provide them explicitly in CLAUDE.md.
</Aside>

## You Are the Main Thread

Think of yourself as a CPU scheduler. Claude Code instances are worker threads. You don't write the code — you **orchestrate** the work.

```
┌─────────────────────────────────────────┐
│          YOU (Main Thread)              │
│  ┌────────────────────────────────────┐ │
│  │  Responsibilities:                 │ │
│  │  • Define tasks and priorities     │ │
│  │  • Allocate context budgets        │ │
│  │  • Review outputs                  │ │
│  │  • Make architectural decisions    │ │
│  │  • Handle exceptions/escalations   │ │
│  └────────────────────────────────────┘ │
│         │          │          │         │
│    ┌────▼───┐ ┌────▼───┐ ┌────▼───┐    │
│    │Worker 1│ │Worker 2│ │Worker 3│    │
│    │(Claude)│ │(Claude)│ │(Claude)│    │
│    │Feature │ │Tests   │ │Review  │    │
│    └────────┘ └────────┘ └────────┘    │
└─────────────────────────────────────────┘
```

**Implications**:
- **Don't write code** when Claude can. Your time is for decisions, not keystrokes.
- **Don't micromanage**. Give clear instructions, then review results.
- **Context-switch deliberately**. Like a scheduler, batch similar tasks.
- **Escalate to yourself**. When Claude is stuck, step in — then hand back.

## From Chatbot to Context System

The most common mistake is treating Claude Code like a chatbot — typing ad-hoc requests and hoping for good output. What separates casual usage from production workflows is a **shift in thinking**:

> **Chatbot mode**: You write good prompts. **Context system**: You build structured context that makes every prompt better.

### Four Layers of Persistent Context

Claude Code has four layers of persistent context that compound over time:

| Layer | What It Does | When to Set Up |
|-------|-------------|----------------|
| **CLAUDE.md** | Persistent rules, conventions, project knowledge | Week 1 |
| **Skills** | Reusable knowledge modules for consistent workflows | Week 2 |
| **Hooks** | Automated guardrails (lint, security, formatting) | Week 2-3 |
| **Project memory** | Cross-session decisions and architectural context | Ongoing |

These are not independent features. They are layers of the **same system**:

- **CLAUDE.md** teaches Claude *what* your project needs
- **Skills** teach Claude *how* to perform specific workflows
- **Hooks** enforce *guardrails* automatically
- **Memory** preserves *decisions* across sessions

### Before vs. After

<Tabs>
  <TabItem label="Before (Chatbot Mode)">
    > "Use pnpm, not npm. And remember our naming convention is..."
    >
    > *(Every session. Every time. Copy-pasting context.)*
  </TabItem>
  <TabItem label="After (Context System)">
    > CLAUDE.md loads conventions automatically. Skills ensure consistent workflows. Hooks enforce quality with zero manual effort. Memory carries decisions forward.
  </TabItem>
