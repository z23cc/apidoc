## What is Claude-Mem?

Claude-Mem is a **persistent memory compression system** plugin for Claude Code. It automatically captures tool usage observations, generates semantic summaries, and makes them available to future sessions — enabling Claude to maintain continuity of knowledge about projects even after sessions end or reconnect.

<CardGrid>
  <Card title="Persistent Memory" icon="open-book">
    Context automatically saved and restored across sessions — Claude never "forgets."
  </Card>
  <Card title="Fully Automatic" icon="approve-check-circle">
    No manual intervention required. Automatically captures, processes, and injects context.
  </Card>
  <Card title="Privacy Control" icon="warning">

  <Card title="Multilingual" icon="translate">

## Quick Install

<Steps>
1. Run the following commands in your Claude Code terminal:
   ```bash
   /plugin marketplace add thedotmack/claude-mem
   /plugin install claude-mem
   ```

2. Restart Claude Code. Context from previous sessions will automatically appear in new sessions.
</Steps>

<Aside type="caution">
Claude-Mem is published on npm, but `npm install -g claude-mem` installs the SDK/library only — it does not register plugin hooks or start the worker service. Always install via the `/plugin` commands above.
</Aside>

### System Requirements

| Component | Requirement |
|-----------|-------------|
| Node.js | 18.0.0 or higher |
| Claude Code | Latest version with plugin support |
| Bun | JavaScript runtime (auto-installed if missing) |
| SQLite 3 | Persistent storage (bundled) |

### Install from Source

```bash
git clone https://github.com/thedotmack/claude-mem.git
cd claude-mem
npm install
npm run build
npm run worker:start    # Manual start (usually auto-starts)
npm run worker:status   # Check status
```

### Data Directory

Data is stored in `~/.claude-mem/`:

| File | Description |
|------|-------------|
| `claude-mem.db` | SQLite database |
| `.worker.pid` | PID file |
| `.worker.port` | Port file |
| `logs/worker-*.log` | Log files |
| `settings.json` | Configuration |

Override with environment variable: `export CLAUDE_MEM_DATA_DIR=/custom/path`

## How It Works

```
┌──────────────────────────────────────────────────────┐
│ Session Start → Inject context from last 10 sessions │
└──────────────────────────────────────────────────────┘
                         ↓
┌──────────────────────────────────────────────────────┐
│ User Prompts → Create session, save user prompts     │
└──────────────────────────────────────────────────────┘
                         ↓
┌──────────────────────────────────────────────────────┐
│ Tool Executions → Capture observations               │
└──────────────────────────────────────────────────────┘
                         ↓
┌──────────────────────────────────────────────────────┐
│ Worker Processes → Extract learnings via Agent SDK   │
└──────────────────────────────────────────────────────┘
                         ↓
┌──────────────────────────────────────────────────────┐
│ Session Ends → Generate summary, ready for next      │
└──────────────────────────────────────────────────────┘
```

### Core Components

| Component | Description |
|-----------|-------------|
| 4 Lifecycle Hooks | SessionStart, UserPromptSubmit, PostToolUse, Stop |
| Smart Install | Cached dependency checker (pre-hook script) |
| Worker Service | HTTP API on port 37777 managed by Bun |
| SQLite Database | Stores sessions, observations, summaries with FTS5 search |
| MCP Search Tools | Query historical context with natural language |
| Web Viewer UI | Real-time visualization with SSE and infinite scroll |

### What Gets Captured

Every time Claude uses a tool, claude-mem captures and extracts:

| Field | Description |
|-------|-------------|
| Title | Brief description of what happened |
| Narrative | Detailed explanation |
| Facts | Key learnings as bullet points |
| Concepts | Relevant tags and categories |
| Type | Classification (decision, bugfix, feature, etc.) |
| Files | Which files were read or modified |

### Session Summaries

When Claude finishes responding (triggering the Stop hook), a summary is automatically generated with:

- **Request** — What you asked for
- **Investigated** — What Claude explored
- **Learned** — Key discoveries and insights
- **Completed** — What was accomplished
- **Next Steps** — What to do next

## Progressive Disclosure

Context injection uses a three-layer progressive disclosure strategy for efficient token usage:

### Layer 1: Index Display (Session Start)

Shows observation titles with token cost estimates, grouped chronologically.

**Token cost**: ~50-200 tokens

### Layer 2: On-Demand Details (MCP Tools)

Ask naturally: "What bugs did we fix?" or "How did we implement X?"

Claude auto-invokes MCP search tools to fetch full details.

**Token cost**: ~100-500 tokens per observation

### Layer 3: Perfect Recall (Code Access)

Read source files directly when full information is needed.

## MCP Search Tools

Claude-Mem provides 4 MCP tools following a token-efficient 3-layer workflow:

### The 3-Layer Workflow

```
Search → Get compact index with IDs (~50-100 tokens/result)
    ↓
Timeline → Get context around interesting results
    ↓
Get Observations → Fetch full details ONLY for filtered IDs
```

Achieves ~**10x token savings** compared to traditional RAG approaches.

### search — Search Memory Index

Search your memory and get a compact index with IDs.

```bash
search(query="authentication bug", type="bugfix", limit=10)
```

**Parameters**: `query`, `limit`, `offset`, `type` (bugfix/feature/decision/discovery/refactor/change), `project`, `dateStart`, `dateEnd`, `orderBy`

### timeline — Get Chronological Context

Get a chronological view of observations around a specific point or query.

```bash
timeline(anchor=12345, depth_before=5, depth_after=5)
# Or search-based
timeline(query="implemented JWT auth", depth_before=3, depth_after=3)
```

### get_observations — Fetch Full Details

Fetch complete observation details by IDs. Always batch multiple IDs in a single call.

```bash
get_observations(ids=[123, 456, 789])
```

### Common Use Cases

**Debugging Issues**:
```
1. search(query="error database connection", type="bugfix", limit=10)
2. timeline(anchor=312, depth_before=3, depth_after=3)
3. get_observations(ids=[312, 489])
```

**Understanding Decisions**:
```
1. search(query="authentication", type="decision", limit=5)
2. get_observations(ids=[<relevant_ids>])
```

## Private Tags

Use `<private>` tags for fine-grained control over what doesn't get persisted.

### Usage

```xml
<private>
This content will not be stored in memory
</private>
```

Claude can see and use this content during the current session, but it won't be saved as an observation.

### Use Cases

**Sensitive Information**:
```xml
Please analyze this error:
<private>
Error: Database connection failed
Host: internal-db-prod.company.com
User: admin_user
</private>
What might be causing this?
```

**Temporary Context**:
```xml
<private>
Background: This is a hotfix for production, deadline is tomorrow
</private>
Help me fix this bug quickly.
```

### Technical Details

| Filtered | Not Filtered |
|----------|--------------|
| User prompt storage | Session summaries |
| Tool input parameters | Claude's responses |
| Tool response output | |
| All searchable content | |

<Aside>
Tags are stripped during storage, not from the live conversation. Claude sees full content including &lt;private&gt; tags during the session.
</Aside>

## Folder Context Files

Claude-Mem can automatically generate `CLAUDE.md` files in project folders, providing directory-level context.

<Aside type="caution">
This feature is disabled by default. Enable it in settings.
</Aside>

### Enable

Edit `~/.claude-mem/settings.json`:

```json
{
  "CLAUDE_MEM_FOLDER_CLAUDEMD_ENABLED": "true"
}
```

### Generated Content

Each folder's `CLAUDE.md` contains an activity timeline:

```markdown
<claude-mem-context>
# Recent Activity

### Jan 4, 2026

| ID | Time | T | Title | Read |
|----|------|---|-------|------|
| #1234 | 4:30 PM | 🔵 | Implemented user authentication | ~250 |
| #1235 | " | 🔴 | Fixed login redirect bug | ~180 |
</claude-mem-context>
```

**User content preservation**: Content outside `<claude-mem-context>` tags is preserved during regeneration.

**Project root exclusion**: Directories containing `.git` are excluded to avoid overwriting hand-written project documentation.

### Cleanup Mode

```bash
# Preview (dry run)
bun scripts/regenerate-claude-md.ts --clean --dry-run

# Actually clean
bun scripts/regenerate-claude-md.ts --clean
```

## Memory Export/Import

Share knowledge across claude-mem installations with automatic duplicate prevention.

### Export

```bash
# Export Windows-related memories
npx tsx scripts/export-memories.ts "windows" windows-memories.json

# Export bug fixes
npx tsx scripts/export-memories.ts "bugfix" bugfixes.json

# Filter by project
npx tsx scripts/export-memories.ts "auth" auth.json --project=myapp
```

### Import

```bash
npx tsx scripts/import-memories.ts windows-memories.json
```

Re-importing automatically skips existing records.

### Safety Features

- Duplicate prevention — won't re-import existing records
- Transactional — all-or-nothing imports
- Read-only export — export script opens database in read-only mode
- Dependency ordering — sessions imported before observations/summaries

## Manual Recovery

Recover observations stuck in the processing queue after worker crashes or system restarts.

### CLI Tool

```bash
# Interactive check and recovery
bun scripts/check-pending-queue.ts

# Auto-process without prompts
bun scripts/check-pending-queue.ts --process

# Limit to 5 sessions
bun scripts/check-pending-queue.ts --process --limit 5
```

### HTTP API

```bash
# Check queue status
curl http://localhost:37777/api/pending-queue

# Trigger recovery
curl -X POST http://localhost:37777/api/pending-queue/process \
  -H "Content-Type: application/json" \
  -d '{"sessionLimit": 10}'
```

### Queue States

| State | Description |
|-------|-------------|
| pending | Queued, waiting to process |
| processing | Currently being processed by SDK agent |
| processed | Completed successfully |
| failed | Failed after 3 retry attempts |

Messages in `processing` state for more than 5 minutes are considered stuck and automatically reset to `pending`.

## Mode System

Claude-Mem uses a flexible mode system to adapt behavior and language.

```json
{
  "CLAUDE_MEM_MODE": "code--zh"
}
```

### Available Modes

| Mode | ID | Description |
|------|-----|-------------|

| Code Chill | `code--chill` | Fewer observations, only records important items |

### Multilingual Code Modes

Use `code--language_code` format to switch languages:

| Language | Mode ID |
|----------|---------|
| Chinese | `code--zh` |
| Japanese | `code--ja` |
| Korean | `code--ko` |

| Russian | `code--ru` |

28 languages supported in total. Modes use the `--` separator for inheritance — child modes override specific parent mode settings.

## Endless Mode (Beta)

Experimental biomimetic memory architecture that dramatically extends Claude's context retention.

<Aside type="caution">
Endless Mode is experimental and not included in the stable release. You must manually switch to the beta branch.
</Aside>

### The Problem

In standard Claude Code sessions, tool outputs accumulate in the context window. After ~50 tool uses, the context fills up (~200k tokens), forcing a new session. This is O(N&sup2;) complexity.

### How It Works

Endless Mode uses a two-tier memory system:

| Tier | Storage | Content |
|------|---------|---------|
| Working Memory | Context window | Compressed observations (~500 tokens each) |
| Archive Memory | Disk transcript | Full tool outputs (searchable) |

After each tool use, it blocks until the worker generates a compressed observation, replacing the full output. This transforms O(N&sup2;) scaling into O(N).

### How to Enable

```bash
cd ~/.claude/plugins/marketplaces/thedotmack/
git checkout beta/endless-mode
npm install
npm run worker:restart
```

## Troubleshooting

### Worker Service Not Starting

```bash
npm run worker:status     # Check status
npm run worker:logs       # View logs
npm run worker:restart    # Restart
lsof -i :37777           # Check port usage
```

### Web Viewer Not Loading

```bash
curl http://localhost:37777/health    # Verify health
npm run worker:restart               # Restart worker
```

### No Context Appearing

- Ensure worker is running: `npm run worker:status`
- Visit `http://localhost:37777` to check for observations
- Verify your API key is configured correctly

### Install Cache Issues

```bash
rm ~/.claude/plugins/marketplaces/thedotmack/.install-version
cd ~/.claude/plugins/marketplaces/thedotmack && npm install --force
```
