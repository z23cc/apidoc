Claude Code is an **agentic assistant** that runs in your terminal. While it excels at coding, it can help with anything you can do from the command line: writing docs, running builds, searching files, researching topics, and more.

This guide covers the core architecture, built-in capabilities, and tips for working effectively.

---

## The Agentic Loop

When you give Claude a task, it works through three phases: **gather context**, **take action**, and **verify results**. These phases blend together. Claude uses tools throughout — searching files to understand your code, editing to make changes, or running tests to check its work.

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ Gather Context   │ ──→ │  Take Action     │ ──→ │ Verify Results   │
│ Search, Read     │     │  Edit, Execute   │     │  Test, Check     │
└─────────────────┘     └─────────────────┘     └─────────────────┘
       ↑                                                │
       └──────────── Adjust based on results ───────────┘
```

The loop adapts to what you ask:
- **Code question** — might only need context gathering
- **Bug fix** — cycles through all three phases repeatedly
- **Refactor** — might involve extensive verification

<Aside type="tip">
You're part of this loop too. You can interrupt at any point to steer Claude in a different direction, provide additional context, or ask it to try a different approach.
</Aside>

### Two Core Components

The agentic loop is powered by two components:

| Component | Role |
|-----------|------|
| **Models** | Reason about code and understand tasks |
| **Tools** | Take action (read files, edit, run commands, etc.) |

Claude Code serves as the **agentic harness** around Claude: it provides the tools, context management, and execution environment that turn a language model into a capable coding agent.

---

## Models

Claude Code uses Claude models to understand your code and reason about tasks. Claude can read code in any language, understand how components connect, and figure out what needs to change to accomplish your goal. For complex tasks, it breaks work into steps, executes them, and adjusts based on what it learns.

### Available Models

Multiple models are available with different tradeoffs:

| Model | Strengths | Best For |
|-------|-----------|----------|
| **Sonnet** | Balanced speed and capability | Most coding tasks |
| **Opus** | Strongest reasoning | Complex architectural decisions |
| **Haiku** | Fastest | Quick, simple tasks |

Switch models:
- During a session: `/model` command
- At launch: `claude --model <name>`

## Tools

Tools are what make Claude Code agentic. Without tools, Claude can only respond with text. With tools, Claude can **act**: read your code, edit files, run commands, search the web, and interact with external services.

### Built-in Tool Categories

| Category | What Claude Can Do |
|----------|-------------------|
| **File operations** | Read files, edit code, create new files, rename and reorganize |
| **Search** | Find files by pattern, search content with regex, explore codebases |
| **Execution** | Run shell commands, start servers, run tests, use git |
| **Web** | Search the web, fetch documentation, look up error messages |
| **Code intelligence** | See type errors and warnings after edits, jump to definitions, find references |

### Tool Usage Example

When you say "fix the failing tests," Claude might:

<Steps>
1. Run the test suite to see what's failing
2. Read the error output
3. Search for the relevant source files
4. Read those files to understand the code
5. Edit the files to fix the issue
6. Run the tests again to verify
</Steps>

Each tool use gives Claude new information that informs the next step. This is the agentic loop in action.

### Extending Capabilities

The built-in tools are the foundation. You can extend with:

| Extension | Description |
|-----------|-------------|
| **Skills** | Extend Claude's knowledge and workflows |
| **MCP** | Connect to external services |
| **Hooks** | Automate workflows |
| **Subagents** | Delegate tasks to independent agents |

## What Claude Can Access

When you run `claude` in a directory, Claude Code gains access to:

<CardGrid>
  <Card title="Your Project" icon="open-book">
    Files in your directory and subdirectories, plus files elsewhere with your permission.
  </Card>
  <Card title="Your Terminal" icon="laptop">
    Any command you could run: build tools, git, package managers, system utilities, scripts.
  </Card>
  <Card title="Git State" icon="github">
    Current branch, uncommitted changes, and recent commit history.
  </Card>
  <Card title="CLAUDE.md" icon="document">
    A markdown file where you store project-specific instructions, conventions, and context.
  </Card>
</CardGrid>

Also includes:
- **Auto memory** — Learnings Claude saves automatically as you work (first 200 lines of MEMORY.md loaded at session start)
- **Configured extensions** — MCP servers, skills, subagents, Claude in Chrome

<Aside>
Because Claude sees your whole project, it can work across it. When you ask "fix the authentication bug," it searches for relevant files, reads multiple files for context, makes coordinated edits, runs tests, and commits changes — unlike inline code assistants that only see the current file.
</Aside>

## Environments and Interfaces

The agentic loop, tools, and capabilities are the same everywhere you use Claude Code. What changes is **where code executes** and **how you interact**.

### Execution Environments

| Environment | Where Code Runs | Use Case |
|-------------|----------------|----------|
| **Local** | Your machine | Default. Full access to files, tools, and environment |
| **Cloud** | Anthropic-managed VMs | Offload tasks, work on repos you don't have locally |
| **Remote Control** | Your machine, browser-controlled | Use the web UI while keeping everything local |

### Interfaces

You can access Claude Code through:

- Terminal (command line)
- Desktop app
- IDE extensions (VS Code, JetBrains)
- claude.ai/code
- Remote Control
- Slack
- CI/CD pipelines

The interface determines how you see and interact with Claude, but the underlying agentic loop is identical.

## Working with Sessions

### Session Basics

Claude Code saves your conversation locally as you work. Every message, tool use, and result is stored, enabling **rewinding**, **resuming**, and **forking** sessions. Before Claude makes code changes, it also snapshots affected files for reverting.

<Aside type="caution">
Sessions are independent. Each new session starts with a fresh context window, without conversation history from previous sessions. Claude persists learnings across sessions using **auto memory**, and you can add persistent instructions in **CLAUDE.md**.
</Aside>

### Working Across Branches

Each Claude Code conversation is a session tied to your current directory. When you resume, you only see sessions from that directory.

- When switching branches, Claude sees the new branch's files but conversation history stays the same
- Use **git worktrees** to create separate directories for branches, enabling parallel Claude sessions

### Resume or Fork

```bash
# Resume last session
claude --continue

# Choose a specific session to resume
claude --resume

# Fork session (new session ID, preserves conversation history)
claude --continue --fork-session
```

<Aside>
If you resume the same session in multiple terminals, messages from both get interleaved. For parallel work from the same starting point, use `--fork-session` to give each terminal its own clean session.
</Aside>

## The Context Window

Claude's context window holds: conversation history, file contents, command outputs, CLAUDE.md, loaded skills, and system instructions. As you work, context fills up.

### When Context Fills Up

Claude Code manages context automatically:

<Steps>
1. Clears older tool outputs first
2. Summarizes the conversation if needed
3. Preserves your requests and key code snippets
4. Detailed instructions from early in the conversation may be lost
</Steps>

**Key advice**: Put persistent rules in **CLAUDE.md** rather than relying on conversation history.

### Context Management Commands

| Command | Description |
|---------|-------------|
| `/context` | See what's using context space |
| `/compact` | Manually compact context |
| `/compact focus on API changes` | Compact with a focus |
| `/mcp` | Check per-MCP-server context costs |

### Managing Context with Skills and Subagents

- **Skills load on demand** — Claude sees skill descriptions at session start, but full content only loads when used. Set `disable-model-invocation: true` to keep descriptions out of context
- **Subagents get fresh context** — completely separate from your main conversation, preventing context bloat. They return a summary when done. This isolation is why subagents help with long sessions

## Safety: Checkpoints and Permissions

Claude has two safety mechanisms: **checkpoints** to undo file changes, and **permissions** to control what Claude can do.

### Checkpoints

Every file edit is reversible. Before Claude edits any file, it snapshots the current contents.

- Press **Esc twice** to rewind to a previous state
- Or ask Claude to undo

<Aside type="caution">
Checkpoints are local to your session, separate from git. They only cover file changes. Actions affecting remote systems (databases, APIs, deployments) can't be checkpointed — that's why Claude asks before running commands with external side effects.
</Aside>

### Permission Modes

Press **Shift+Tab** to cycle through permission modes:

| Mode | Description |
|------|-------------|
| **Default** | Claude asks before file edits and shell commands |
| **Auto-accept edits** | Claude edits files without asking, still asks for commands |
| **Plan mode** | Claude uses read-only tools only, creating a plan you approve first |

You can also allow specific commands in `.claude/settings.json` (like `npm test` or `git status`). Settings can be scoped from organization-wide policies down to personal preferences.

## Tips for Working Effectively

### Ask Claude Code for Help

Claude Code can teach you how to use it. Ask questions like:

- "How do I set up hooks?"
- "What's the best way to structure my CLAUDE.md?"

Built-in commands also guide you:

| Command | Description |
|---------|-------------|
| `/init` | Walk through creating a CLAUDE.md for your project |
| `/agents` | Configure custom subagents |
| `/doctor` | Diagnose common installation issues |

### It's a Conversation

Claude Code is conversational. You don't need perfect prompts. Start with what you want, then refine:

```
> Fix the login bug

[Claude investigates, tries something]

> That's not quite right. The issue is in the session handling.

[Claude adjusts approach]
```

When the first attempt isn't right, you don't start over. You **iterate**.

### Interrupt and Steer

You can interrupt Claude at any point. If it's going down the wrong path, type your correction and press Enter. Claude will stop what it's doing and adjust based on your input. No need to wait or start over.

### Be Specific Upfront

The more precise your initial prompt, the fewer corrections you'll need. Reference specific files, mention constraints, and point to example patterns:

```
> The checkout flow is broken for users with expired cards.
> Check src/payments/ for the issue, especially token refresh.
> Write a failing test first, then fix it.
```

Vague prompts like "fix the login bug" work, but you'll spend more time steering.

### Give Claude Something to Verify Against

Claude performs better when it can check its own work. Include test cases, paste screenshots, or define expected output:

```
> Implement validateEmail. Test cases: 'user@example.com' → true,
> 'invalid' → false, 'user@.com' → false. Run the tests after.
```

### Explore Before Implementing

For complex problems, separate research from coding. Use **plan mode** (Shift+Tab twice) to analyze the codebase first:

```
> Read src/auth/ and understand how we handle sessions.
> Then create a plan for adding OAuth support.
```

Review the plan, refine through conversation, then let Claude implement. This two-phase approach produces better results than jumping straight to code.

### Delegate, Don't Dictate

Think of delegating to a capable colleague. Give context and direction, then trust Claude to figure out the details:

```
> The checkout flow is broken for users with expired cards.
> The relevant code is in src/payments/. Can you investigate and fix it?
```

You don't need to specify which files to read or what commands to run. Claude figures that out.
