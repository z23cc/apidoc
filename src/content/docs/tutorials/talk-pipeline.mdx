## 概述

将原始文章、转录稿或笔记转化为一场完整的会议演讲 -- 包括可直接用于 Kimi AI 生成幻灯片的提示词。六个阶段，两种模式，一个人工审核检查点。

<CardGrid>
  <Card title="六阶段流水线" icon="list-format">
    从提取到修订，完整覆盖演讲准备全流程。
  </Card>
  <Card title="两种模式" icon="setting">
    REX（经验回顾，带 git 历史和指标）和 Concept（想法/论点导向）。
  </Card>
  <Card title="AI 到 AI 交接" icon="rocket">
    Claude 生成叙事结构，Kimi 生成视觉幻灯片。
  </Card>
  <Card title="人工审核" icon="approve-check-circle">
    关键决策点必须由你做出，确保演讲方向正确。
  </Card>
</CardGrid>

```

        |
        |──────────────────────────────────┐
        |                                  |
        v（仅 REX）                        v
[阶段 2] 研究                  [阶段 3] 概念
  git-archaeology.md            concepts.md
  changelog-analysis.md         concepts-enriched.md
  timeline.md                       |
        |                           |
        |──────────────┬────────────┘
                       |
                       v
              [阶段 4] 定位
                angles.md
                titre.md
                descriptions.md
                feedback-draft.md
                       |
                  [检查点]
             用户选择角度 + 标题
                       |
                       v
              [阶段 5] 脚本
                pitch.md

                       |
                       v
             [阶段 6] 修订
              revision-sheets.md
```

## 适用场景

此流水线适用于以下情况：

<Tabs>
  <TabItem label="REX 模式">
    你构建了某样东西、已经上线、拥有真实指标 -- 想将其转化为结构化的会议演讲。

    **额外需要**：访问你要分析的 git 仓库
  </TabItem>
  <TabItem label="Concept 模式">
    你有一篇文章、笔记或想法，想将其转化为结构化的演示文稿。

    **自动跳过**阶段 2（研究）
  </TabItem>
</Tabs>

**适合的格式**：
- 会议演讲（20-45 分钟）
- Meetup 演示（15-30 分钟）
- 内部技术分享
- 工作坊开场

**不适合**：经常性会议的幻灯片更新、10 分钟以下的闪电演讲、没有现有素材可使用的情况。

## 前置条件

| 类别 | 要求 |
|------|------|
| **必需** | 安装了 skills 的 Claude Code |
| **必需** | 源素材：`.mdx` 文章、`.md` 转录稿、笔记或混合 |
| **必需** | 演讲元数据：slug、活动名称、日期、时长、受众描述 |
| **REX 模式** | 访问要分析的 git 仓库 |
| **幻灯片生成** | [kimi.com](https://kimi.com) 免费账户（无需 API -- 只需复制粘贴） |
| **推荐** | 项目根目录下的 `talks/` 目录用于输出文件 |

## 分阶段详解

### 阶段 1：提取

读取源素材，产出结构化摘要 -- 叙事弧线、关键指标、主要主题、空白点。

**调用方式**：
```
/talk-stage1-extract
```

或通过编排器：
```
/talk-pipeline --stage=extract --slug=my-talk --event="Conf 2026" --date=2026-06-15 --duration=30 --audience="senior devs"
```

<Aside type="tip" title="继续前检查">
- 叙事弧线连贯（不是泛泛的"这个演讲关于 AI..."）
- 所有指标都有明确来源
- 空白点已列出
- 类型检测正确（REX / Concept / Hybrid）
</Aside>

---

### 阶段 2：研究（仅 REX 模式）

Git 考古 -- 提取速度指标、交叉参考 CHANGELOG、构建经 git 验证的事实时间线。

**使用的 Git 命令**（只读）：
```bash
git log --pretty=format:"%Y-%m" | sort | uniq -c   # 按月速度
git shortlog -sn --no-merges                         # 贡献者
git tag --sort=version:refname                       # 发布版本
git log --merges --oneline | wc -l                   # 已合并 PR
```

<Aside type="caution">
仅使用只读 git 命令 -- 永不修改仓库。在 git 中未找到的日期标记为"未验证" -- 从不估算。
</Aside>

---

### 阶段 3：概念

构建编号、评分的概念目录。每个概念获得演讲潜力评分。

**评分标准**：

| 级别 | 标准 |
|------|------|
| **HIGH** | 可展示、反直觉、有验证数据、30 秒内可操作 |
| **MEDIUM** | 有用但在预期范围内，缺少证据或数据 |
| **LOW** | 太抽象、已被充分讨论、难以在幻灯片中展示 |

**评分纪律**：最多 30% 为 HIGH -- 选择性是重点。

---

### 阶段 4：定位 + 检查点

生成战略角度、标题、描述和同行反馈草稿。然后**停下来等待**你选择角度和标题。

**生成的内容**：
- **3-4 个角度**：各含优势、弱点、受众适配度、评分（/5）
- **推荐**：一个明确选择，附结构化理由
- **每个角度 3-5 个标题**
- **短描述**（~100 字摘要）+ **长描述**（~250 字 CFP）
- **反馈草稿**：可直接发送的同行验证消息

<Aside type="caution" title="强制检查点">
阶段 5 在没有明确人工确认的情况下**不能开始**。角度和标题决定了后续的一切 -- 五幕结构、哪些概念浮出水面、Kimi 提示词的风格。自动选择会产出一场平庸的演讲。
</Aside>

**检查点操作**：
1. 阅读 `angles.md` -- 不要跳过，推荐可能对你的上下文不适用
2. 可选地将 `feedback-draft.md` 发送给信任的同行
3. 回复你的选择（可以是推荐、修改或完全不同的方案）

---

### 阶段 5：脚本

构建五幕完整演讲（含演讲者笔记）、幻灯片规格和 Kimi 提示词。

**三个交付物**：

1. **`pitch.md`** -- 五幕叙事，含演讲者笔记、计时、关键时刻
2. **`slides.md`** -- 逐张幻灯片规格：标题、视觉描述、关键文字（&le;30 字）、演讲者笔记、时长
3. **`kimi-prompt.md`** -- [kimi.com](https://kimi.com) 完整提示词，含设计系统、配色方案、排版规格

**一张幻灯片一个想法规则**：如果描述一张幻灯片需要用"和"，就拆分它。

---

### 阶段 6：修订

生成修订工作表 -- 按幕快速导航、主概念表、Q&A 备忘单、术语表。

**修订工作表包含**：
- 按幕的锚点链接导航
- 每幕分解：关键概念 + 指标 + 轶事 + 可能的 Q&A
- 主表格：概念 -> 1-2 句定义 -> 可分享的 URL
- Q&A 备忘单：6-10 个可能的问题及简短回答（&le;20 秒可讲完）
- 指标汇总：所有数字集中一处
- 术语表：技术术语，每个最多 10 字

## Kimi 交接

阶段 5 生成 `{slug}-kimi-prompt.md` -- 用于 [kimi.com](https://kimi.com) 的完整提示词。

<Steps>

1. 打开生成的文件
2. 确认没有遗留的占位符（搜索文件）
3. 前往 [kimi.com](https://kimi.com) -- 免费账户，无需 API
4. 开始新对话
5. 复制粘贴整个提示词
6. Kimi 生成演示文稿（PowerPoint 或 PDF）

</Steps>

**内嵌设计系统**：
- 深色主题（#0a0a0a 背景）
- 橙色强调（#f97316）用于关键数字和 CTA
- Inter/SF Pro 排版
- 每张幻灯片最多 30 字，数字为主角
- WCAG AA 对比度（投影仪安全）

## 适配不同场景

<Tabs>
  <TabItem label="闪电演讲（10-15 分钟）">
    - 保留阶段 1、3、4、5 -- 即使 REX 模式也跳过阶段 2
    - 阶段 3：限制 8-10 个概念（只留 HIGH）
    - 阶段 4：最多生成 2 个角度
    - 阶段 5：目标 ~8-10 张幻灯片
    - 跳过阶段 6
  </TabItem>
  <TabItem label="45 分钟演讲">
    - 完整流水线，所有阶段
    - 阶段 3：目标 25-35 个概念
    - 阶段 5：20-25 张幻灯片
    - 阶段 6 变得关键（Q&A 持续 10-15 分钟）
  </TabItem>
  <TabItem label="工作坊（90+ 分钟）">
    - 运行流水线至阶段 4
    - 阶段 5：将 slides.md 替换为活动规格（练习、计时、分组）
    - Kimi 提示词部分变为可选
  </TabItem>
</Tabs>

## 常见陷阱

<CardGrid>
  <Card title="无来源的指标" icon="warning">
    阶段 1 提取指标但不验证。阶段 2 验证。Concept 模式中任何你提到的指标必须有明确来源 -- 否则删除。
  </Card>
  <Card title="幻灯片过载" icon="warning">
    Kimi 提示词强制 30 字/张，但 pitch.md 可能偏向项目列表。测试一张一想法规则。
  </Card>
  <Card title="跳过检查点" icon="warning">
    不验证角度+标题就运行阶段 5 会产出技术上正确但方向错误的脚本。
  </Card>
  <Card title="反馈草稿发送太晚" icon="warning">
    角度/标题阶段的同行反馈可操作。完成脚本后的反馈大多只产生遗憾。
  </Card>
</CardGrid>

## 设计模式亮点

### 基于文件的状态链

每个阶段写入下一阶段读取的文件。状态通过文件系统持久化 -- 无内存耦合。你可以在阶段 2 一周后再运行阶段 3。

### 工具权限范围

阶段 2 是唯一需要 Bash 的阶段（用于 git 命令）。其他阶段仅需 Read + Write。

### AI 到 AI 交接

Claude 不直接生成幻灯片 -- 它生成另一个 AI（Kimi）执行的规格。结合两者的优势：Claude 做结构化叙事推理，Kimi 做视觉演示生成。
