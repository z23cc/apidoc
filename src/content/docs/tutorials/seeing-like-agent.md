构建 Agent 系统最困难的部分之一是构建其动作空间（action space）。

Claude 通过工具调用（Tool Calling）来执行操作，但在 Claude API 中有多种方式可以构建工具，包括 bash、skills 以及最近推出的代码执行等基础能力。

面对这么多选择，你该如何设计 Agent 的工具？是只需要一个像代码执行或 bash 这样的工具？还是需要 50 个工具，每个工具对应 Agent 可能遇到的一个用例？

为了站在模型的角度思考，我喜欢想象自己被给予一道困难的数学题。你会想要什么工具来解决它？这取决于你自己的能力！

纸笔是最基本的，但你会受限于手工计算。计算器会更好，但你需要知道如何操作更高级的功能。最快最强大的选择是计算机，但你必须知道如何使用它来编写和执行代码。

这是设计 Agent 的一个有用框架。你想给它提供与其自身能力相匹配的工具。但你如何知道这些能力是什么？你需要关注、阅读它的输出、进行实验。你需要学会像 Agent 一样思考。

以下是我们在构建 Claude Code 时通过观察 Claude 学到的一些经验。

## 改进提问能力与 AskUserQuestion 工具

在构建 AskUserQuestion 工具时，我们的目标是改进 Claude 提问的能力（通常称为 elicitation）。

虽然 Claude 可以用纯文本提问，但我们发现回答这些问题感觉需要不必要的时间。我们如何降低这种摩擦并增加用户与 Claude 之间的通信带宽？

### 尝试 #1 - 修改 ExitPlanTool

我们首先尝试在 ExitPlanTool 中添加一个参数，在计划旁边包含一个问题数组。这是最容易实现的，但它让 Claude 感到困惑，因为我们同时要求一个计划和一组关于计划的问题。如果用户的答案与计划所说的冲突怎么办？Claude 需要调用 ExitPlanTool 两次吗？我们需要另一种方法。

### 尝试 #2 - 改变输出格式

接下来，我们尝试修改 Claude 的输出指令，使用稍微修改的 markdown 格式来提问。例如，我们可以要求它输出带有方括号中备选项的项目符号问题列表。然后我们可以解析并将该问题格式化为用户界面。

虽然这是我们能做的最通用的改变，Claude 似乎也能很好地输出这种格式，但这并不能保证。Claude 会添加额外的句子、省略选项或使用完全不同的格式。

### 尝试 #3 - AskUserQuestion 工具

最后，我们创建了一个 Claude 可以在任何时候调用的工具，但特别提示它在计划模式下这样做。当工具触发时，我们会显示一个模态框来显示问题，并阻塞 Agent 的循环直到用户回答。

这个工具允许我们提示 Claude 输出结构化内容，并帮助我们确保 Claude 给用户提供多个选项。它还为用户提供了组合此功能的方式，例如在 Agent SDK 中调用它或在 skills 中引用它。

最重要的是，Claude 似乎喜欢调用这个工具，我们发现它的输出效果很好。即使是设计最好的工具，如果 Claude 不理解如何调用它，也不会起作用。

这是 Claude Code 中提问的最终形式吗？我们不确定。正如你将在下一个例子中看到的，对一个模型有效的方法可能不是另一个模型的最佳选择。

## 随能力更新 - Tasks 与 Todos

当我们首次推出 Claude Code 时，我们意识到模型需要一个待办事项列表来保持正轨。待办事项可以在开始时编写，并在模型完成工作时勾选。为此，我们给 Claude 提供了 TodoWrite 工具，它可以编写或更新待办事项并显示给用户。

但即使如此，我们经常看到 Claude 忘记它要做什么。为了适应，我们每 5 轮插入系统提醒，提醒 Claude 它的目标。

但随着模型的改进，它们不仅不需要被提醒待办事项列表，反而会发现它具有限制性。被发送待办事项列表的提醒让 Claude 认为它必须坚持列表而不是修改它。我们还看到 Opus 4.5 在使用子代理方面变得更好，但子代理如何在共享的待办事项列表上协调？

看到这一点，我们用 Task 工具替换了 TodoWrite。待办事项是关于让模型保持正轨，而任务更多的是帮助代理之间相互通信。任务可以包含依赖关系，在子代理之间共享更新，模型可以修改和删除它们。

随着模型能力的提高，你的模型曾经需要的工具现在可能会限制它们。不断重新审视关于需要哪些工具的先前假设很重要。这也是为什么坚持支持一小组具有相当相似能力配置的模型是有用的。

## 设计搜索界面

对于 Claude 来说，特别重要的一组工具是可以用来构建自己上下文的搜索工具。

当 Claude Code 首次推出时，我们使用 RAG 向量数据库为 Claude 查找上下文。虽然 RAG 强大且快速，但它需要索引和设置，并且在各种不同的环境中可能很脆弱。更重要的是，Claude 是被**给予**这个上下文，而不是自己找到上下文。

但如果 Claude 可以在网络上搜索，为什么不搜索你的代码库呢？通过给 Claude 一个 Grep 工具，我们可以让它搜索文件并自己构建上下文。

这是我们看到的一个模式，随着 Claude 变得更聪明，如果给它正确的工具，它在构建自己的上下文方面变得越来越好。

当我们引入 Agent Skills 时，我们正式化了渐进式披露（progressive disclosure）的想法，它允许代理通过探索逐步发现相关上下文。

Claude 可以读取 skill 文件，这些文件可以引用模型可以递归读取的其他文件。事实上，skills 的一个常见用途是为 Claude 添加更多搜索能力，比如给它关于如何使用 API 或查询数据库的指令。

在一年的时间里，Claude 从基本上无法构建自己的上下文，发展到能够跨多层文件进行嵌套搜索以找到它需要的确切上下文。

渐进式披露现在是我们用来添加新功能而不添加工具的常用技术。

## 渐进式披露 - Claude Code 指南 Agent

Claude Code 目前有约 20 个工具，我们不断问自己是否需要所有这些工具。添加新工具的门槛很高，因为这给模型多了一个需要考虑的选项。

例如，我们注意到 Claude 对如何使用 Claude Code 了解不够。如果你问它如何添加 MCP 或斜杠命令的作用，它无法回答。

我们本可以将所有这些信息放在系统提示中，但鉴于用户很少问这个问题，这会增加上下文腐烂并干扰 Claude Code 的主要工作：编写代码。

相反，我们尝试了一种渐进式披露的形式。我们给 Claude 一个指向其文档的链接，然后它可以加载以搜索更多信息。这有效，但我们发现 Claude 会将大量结果加载到上下文中以找到正确答案，而实际上你只需要答案。

所以我们构建了 Claude Code 指南子代理，当你询问 Claude 关于自己的问题时，Claude 会被提示调用它，该子代理有关于如何很好地搜索文档以及返回什么的详细指令。

虽然这并不完美，当你问它如何设置自己时 Claude 仍然会感到困惑，但它比以前好多了！我们能够在不添加工具的情况下向 Claude 的动作空间添加内容。

## 艺术，而非科学

如果你希望得到一套关于如何构建工具的严格规则，不幸的是这不是那种指南。为你的模型设计工具既是一门艺术，也是一门科学。它在很大程度上取决于你使用的模型、Agent 的目标以及它运行的环境。

经常实验，阅读你的输出，尝试新事物。像 Agent 一样思考。
