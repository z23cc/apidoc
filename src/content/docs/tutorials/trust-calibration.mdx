AI 生成的代码需要**按风险等级进行对等验证**。盲目接受所有输出或逐行审查每一行代码都是浪费时间。本教程帮助你校准信任度，建立高效的验证策略。

---

## 验证债务问题

研究表明，AI 生成的代码缺陷率高于人工编写的代码：

| 指标 | AI vs 人工 | 来源 |
|------|-----------|------|
| 逻辑错误 | 多 1.75 倍 | ACM 研究，2025 |
| 安全缺陷 | 45% 包含漏洞 | Veracode GenAI 报告，2025 |
| XSS 漏洞 | 多 2.74 倍 | CodeRabbit 研究，2025 |
| PR 体积增长 | +18% | Jellyfish，2025 |
| 每 PR 事故率 | +24% | Cortex.io，2026 |
| 变更失败率 | +30% | Cortex.io，2026 |

<Aside type="note">
**核心洞察**：AI 生产代码的速度更快，但验证成为了瓶颈。问题不是"它能用吗？"而是"我怎么知道它能用？"
</Aside>

<Aside type="tip">
**关于下游可维护性的补充**：一项随机对照实验（Borg 等，2025，n=151 名专业开发者）发现，下游开发者维护 AI 生成代码与人工代码所需的时间没有显著差异。上述缺陷率是真实的，但它们不会系统性地转化为更高的维护负担。风险范围比通常假设的要窄。
</Aside>

## 验证等级划分

不同类型的代码需要不同程度的审查。将验证力度与风险匹配：

<CardGrid>
  <Card title="模板代码" icon="approve-check-circle">
    **验证等级：** 快速浏览

    配置文件、导入语句等。花 10-30 秒扫一眼，信任整体结构。
  </Card>
  <Card title="工具函数" icon="pencil">
    **验证等级：** 快速测试

    格式化、辅助函数等。花 1-2 分钟跑一个正常路径测试。
  </Card>
  <Card title="业务逻辑" icon="open-book">
    **验证等级：** 深度审查 + 测试

    花 5-15 分钟逐行审查，覆盖边界情况。
  </Card>
  <Card title="安全关键代码" icon="warning">
    **验证等级：** 最高级 + 工具辅助

    认证、加密、输入验证等。花 15-30 分钟，使用静态分析、模糊测试、同行评审。
  </Card>
</CardGrid>

| 代码类型 | 验证等级 | 时间投入 | 技术手段 |
|---------|---------|---------|---------|
| 模板代码（配置、导入） | 快速浏览 | 10-30 秒 | 扫一眼，信任结构 |
| 工具函数（格式化、辅助） | 快速测试 | 1-2 分钟 | 一个正常路径测试 |
| 业务逻辑 | 深度审查 + 测试 | 5-15 分钟 | 逐行审查，边界情况 |
| 安全关键代码（认证、加密） | 最高级 + 工具 | 15-30 分钟 | 静态分析、模糊测试、同行评审 |
| 外部集成（API、数据库） | 集成测试 | 10-20 分钟 | Mock + 真实端点测试 |

## 独立开发者 vs 团队策略

<Tabs>
  <TabItem label="独立开发者">
    没有同行评审时，用以下方式弥补：

    <Steps>

    1. **高测试覆盖率（&gt;70%）**：你的安全网

    2. **快速审查（Vibe Review）**：介于"盲目接受"和"逐行审查"之间的中间层
       - 阅读提交信息/摘要
       - 扫一眼 diff 中是否有意外的文件变更
       - 运行测试
       - 在应用中快速检查
       - 如果通过就发布

    3. **静态分析工具**：ESLint、SonarQube、Semgrep 捕捉你遗漏的问题

    4. **时间约束**：不要花 30 分钟审查一个 10 行的工具函数

    </Steps>

    ```
    独立开发者工作流：
    生成 → 快速审查 → 测试通过？ → 发布
                    ↓
            测试失败？ → 深度审查 → 修复
    ```
  </TabItem>
  <TabItem label="团队协作">
    有多名开发者时：

    <Steps>

    1. **AI 初审**：让 Claude 或 Copilot 先审查（可发现 70-80% 的问题）

    2. **人工签字确认**：AI 审查不等于批准

    3. **关键路径由领域专家审查**：安全代码交给安全培训过的审查者

    4. **轮换审查者**：防止形成审查盲区

    </Steps>

    ```
    团队工作流：
    生成 → AI 审查 → 人工审查 → 合并
              ↓              ↓
         标记问题        最终批准
    ```
  </TabItem>
</Tabs>

## "证明它能用"清单

在发布 AI 生成的代码前，逐项验证：

<Tabs>
  <TabItem label="功能正确性">
    - 正常路径可用（手动测试或自动化）
    - 边界情况已处理（null、空值、边界值）
    - 错误状态优雅处理（没有静默失败）
  </TabItem>
  <TabItem label="安全基线">
    - 输入验证已就位（永远不信任用户输入）
    - 没有硬编码的密钥（搜索 `password`、`secret`、`key`）
    - 认证/授权检查完整（没有绕过现有防护）
  </TabItem>
  <TabItem label="集成检查">
    - 现有测试仍然通过
    - diff 中没有意外的文件变更
    - 新增的依赖是合理的且已审计
  </TabItem>
  <TabItem label="代码质量">
    - 遵循项目规范（命名、结构）
    - 没有明显的性能问题（N+1 查询、内存泄漏）
    - 注释解释"为什么"而非"做什么"
  </TabItem>
</Tabs>

## 需要避免的反模式

| 反模式 | 问题所在 | 更好的做法 |
|--------|---------|-----------|
| "能编译就发布" | 语法正确不等于逻辑正确 | 至少跑一个测试 |
| "AI 写的一定安全" | AI 优化的是合理性，不是安全性 | 安全关键代码始终手动审查 |
| "测试通过就完事" | 测试可能没有覆盖变更的部分 | 检查修改行的测试覆盖率 |
| "跟上次一样" | 上下文变了，AI 可能生成不同的代码 | 每次生成都是独立的 |
| "高级工程师写的提示词" | 资深不等于输出质量有保证 | 审查输出，而非输入 |
| "只是模板代码" | 即使模板代码也可能隐藏问题 | 至少扫一眼看有没有意外 |

## 随时间校准信任度

<Steps>

1. **初始阶段：谨慎对待**

   刚接触 Claude Code 时审查所有内容

2. **追踪失败模式**

   Bug 是从哪里溜进来的？记录问题模式

3. **加强关键路径**

   在有过事故的领域加倍投入审查力度

4. **放松低风险领域**

   对于稳定的、有测试的代码类型，可以更多信任 AI

5. **定期审计**

   偶尔抽查那些"被信任"的代码

</Steps>

<Aside type="tip">
**心理模型**：把 AI 想象成一个能力强的初级开发者。你不会不审查就部署他们的代码，但也不会重写他们写的每一行。
</Aside>
